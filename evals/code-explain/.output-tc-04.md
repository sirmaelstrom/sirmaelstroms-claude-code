## Overview

This is an **O(1) LRU (Least Recently Used) cache** — a fixed-capacity key-value store that evicts the least recently accessed entry when full. The insight that makes it work is combining two data structures: a **hash map** for O(1) lookup and a **doubly linked list** for O(1) ordering. Neither structure alone can do both — this design uses them together.

The list encodes recency: the node just before the tail sentinel is the **most recently used (MRU)**, and the node just after the head sentinel is the **least recently used (LRU)**. Every access promotes a node to the tail end.

---

## Diagram

```
                   LRU ← ─ ─ ─ ─ ─ ─ ─ ─ ─ → MRU
                   
  [head] ←→ [node A] ←→ [node B] ←→ [node C] ←→ [tail]
 (sentinel)                                      (sentinel)

  cache = { "a": nodeA, "b": nodeB, "c": nodeC }
```

The two sentinel nodes (`head`, `tail`) are dummy nodes that never hold real data. They eliminate all "is this the first/last node?" edge cases from `_add` and `_remove` — those methods always have valid neighbors to work with.

```
get("b"):                         put("d") when at capacity:
  nodeB = cache["b"]              
  _remove(nodeB)                  LRU = head.next  → nodeA
  _add(nodeB)   → moves to tail   _remove(nodeA)
  return nodeB.val                del cache["a"]
```

## Breakdown

### Initialization

```python
self.head = Node(0, 0)
self.tail = Node(0, 0)
self.head.next = self.tail
self.tail.prev = self.head
```

Creates an empty doubly linked list with two permanent sentinel nodes. They exist only as anchors — `head.next` always points to the LRU real node (or `tail` if empty), and `tail.prev` always points to the MRU real node.

---

### `_remove(node)` — splice out a node

```python
node.prev.next = node.next
node.next.prev = node.prev
```

A standard doubly linked list removal. The node's two neighbors are connected to each other, cutting the node out. The node itself still holds its `prev`/`next` pointers, but they're stale — the caller is responsible for re-inserting or discarding it.

Before:
```
... ←→ [prev] ←→ [node] ←→ [next] ←→ ...
```
After:
```
... ←→ [prev] ←→ [next] ←→ ...
         (node is orphaned)
```

---

### `_add(node)` — insert before tail (MRU position)

```python
prev = self.tail.prev
prev.next = node
node.prev = prev
node.next = self.tail
self.tail.prev = node
```

This wires in the node as the new MRU. Four pointer assignments. Because the tail sentinel always exists, there's no null check needed — `self.tail.prev` is always valid.

Before:
```
... ←→ [prev] ←→ [tail]
```
After:
```
... ←→ [prev] ←→ [node] ←→ [tail]
```

---

### `get(key)` — read with promotion

```python
if key in self.cache:
    node = self.cache[key]
    self._remove(node)
    self._add(node)
    return node.val
return -1
```

The dict gives us the node in O(1). We then remove it from its current position and re-insert it at the tail — promoting it to MRU. This two-step move is the core of the LRU mechanic. A miss returns `-1` (LeetCode convention).

---

### `put(key, value)` — write with eviction

```python
if key in self.cache:
    self._remove(self.cache[key])
node = Node(key, value)
self._add(node)
self.cache[key] = node
if len(self.cache) > self.capacity:
    lru = self.head.next
    self._remove(lru)
    del self.cache[lru.key]
```

Three distinct cases handled in sequence:

1. **Key exists** — remove the old node (the new one will replace it at MRU position)
2. **Insert** — create a new node, add to tail, register in dict
3. **Over capacity** — evict `head.next`, which is always the LRU node. Note that `lru.key` is used to also remove it from the dict — this is why nodes store both `key` and `val`, not just `val`.

The eviction check happens *after* insertion, which means the cache momentarily holds `capacity + 1` entries before the LRU is removed. This is correct behavior.

## Complexity

| Operation | Time | Why |
|---|---|---|
| `get` | O(1) | Dict lookup + 4 pointer swaps |
| `put` | O(1) | Dict write + 4 pointer swaps + optional eviction (also O(1)) |
| Space | O(capacity) | Dict + list both bounded by capacity |

## Patterns

**Composite data structure** — the standard pattern for LRU caches. The dict and list are co-maintained: every node in the list has a corresponding dict entry pointing to it. The `Node` stores `key` expressly so eviction can update the dict.

**Sentinel nodes** — a classic linked list simplification. Without them, `_remove` would need `if node.prev: node.prev.next = ...` guards for head/tail edge cases. The sentinels mean those cases never arise.

**Promote-on-access** — rather than tracking timestamps or counters, recency is encoded structurally. A node's position *is* its recency score. This is O(1) to update vs. O(n) to re-sort.

## Concerns

**Missing `Node` class** — the snippet doesn't include it, but it must be a doubly linked list node with at least `key`, `val`, `prev`, and `next` attributes. The `prev`/`next` aren't shown initialized to `None` here, which could cause attribute errors if `_remove` is called on a node that was never inserted.

**Thread safety** — dict operations and pointer rewiring are not atomic. Concurrent `get`/`put` calls would cause data corruption. A `threading.Lock` would be needed for multi-threaded use.

**`put` eviction timing** — the cache holds `capacity + 1` nodes for one brief moment during insertion. In a memory-constrained environment this is a subtle but real issue.
