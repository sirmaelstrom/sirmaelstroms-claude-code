## Overview

`rotateSession` manages a sliding-window session lifecycle for a Discord channel. It enforces a message-count cap per session: if the current active session hasn't hit the limit, return it; if it has (or none exists), close it and open a fresh one.

This is the mechanism that keeps Observatory's per-channel conversation context bounded — preventing unbounded context accumulation in the SQLite sessions table.

## Breakdown

**1. Look up the active session for this channel**
```sql
SELECT id, message_count FROM sessions WHERE channel_id = ? AND active = 1
```
Fetches at most one row (enforced by the `active = 1` constraint, which should have at most one live session per channel).

**2. Return early if under the cap**
```typescript
if (current && current.message_count < maxMessages) {
  return current.id;
}
```
Hot path — most calls land here. No writes, just a session ID hand-back to the caller.

**3. Retire the full session (if one exists)**
```typescript
db.prepare('UPDATE sessions SET active = 0, ended_at = ? WHERE id = ?')
  .run(new Date().toISOString(), current.id);
```
Stamps `ended_at` and flips `active` to 0. This preserves the full message history for that session without deleting it.

**4. Create a new session**
```typescript
const newId = crypto.randomUUID();
db.prepare('INSERT INTO sessions ...').run(newId, channelId, new Date().toISOString());
return newId;
```
Fresh UUID, `message_count` starts at 0, `active = 1`. The caller gets the new session ID.
