## Overview

This is a **token bucket rate limiter** — the standard algorithm for allowing bursts up to a ceiling while enforcing a steady-state throughput limit. The bucket holds up to `maxTokens` tokens and refills at `refillRate` tokens/second. Each request consumes tokens; if the bucket runs dry, the request is rejected.

The key design choice: **refill is lazy** — it happens only when `tryConsume` is called, not on a background timer. This is efficient (no scheduler, no concurrency complexity) and correct in Node.js's single-threaded runtime.

---

## Breakdown

**Constructor**

```typescript
this.tokens = maxTokens;        // start full — allows immediate burst
this.lastRefill = Date.now();   // anchor for elapsed-time calculation
```

Starting full means a newly created limiter immediately allows a burst up to `maxTokens`. This is the expected behavior for most APIs (don't penalize fresh clients), but worth knowing if you create limiters mid-traffic.

---

**`refill()` — the core math**

```typescript
const elapsed = (now - this.lastRefill) / 1000;          // ms → seconds
this.tokens = Math.min(
  this.maxTokens,
  this.tokens + elapsed * this.refillRate                 // proportional top-up
);
this.lastRefill = now;
```

This converts wall-clock elapsed time into token credit. Example: `refillRate = 10`, `elapsed = 0.5s` → add 5 tokens. `Math.min` caps the bucket so time doesn't accumulate infinitely while the limiter is idle.

`lastRefill` is always updated to `now`, even if no tokens were needed — this prevents double-counting time on the next call.

---

**`tryConsume(cost)`**

```typescript
this.refill();                    // always refill first
if (this.tokens >= cost) {
  this.tokens -= cost;
  return true;
}
return false;                     // reject without consuming
```

Non-blocking — returns immediately. The caller decides what to do on `false` (drop, queue, retry after delay). Partial consumption never happens: either the full `cost` is paid or nothing is touched.

---

## Complexity

| Metric | Value |
|---|---|
| Time complexity | O(1) — all operations are arithmetic |
| Space complexity | O(1) — two numbers per limiter instance |
| Cyclomatic complexity | 2 (one branch in `tryConsume`) |

No timers, no queues, no allocations at call time.

---

## Patterns

- **Token Bucket** — the canonical burst-tolerant rate limiting algorithm. Distinguished from leaky bucket (which smooths output) and fixed-window/sliding-window (which have boundary spike problems).
- **Lazy evaluation** — state is computed on demand rather than maintained continuously. Correct in single-threaded environments; would need synchronization in a multi-threaded context.
- **Guard clause with early return** — `tryConsume` follows the fail-fast pattern; no mutation on failure.

---

## Concerns

**Clock regression** — `Date.now()` can go backwards if the system clock is corrected (NTP step). A negative `elapsed` drains tokens. Fixable with a floor:

```typescript
const elapsed = Math.max(0, (now - this.lastRefill) / 1000);
```

**Unvalidated `cost > maxTokens`** — if `cost` exceeds `maxTokens`, `tryConsume` will always return `false`. No request can ever succeed. No error, no diagnostic — it silently starves the caller. Consider:

```typescript
if (cost > this.maxTokens) throw new Error(`cost ${cost} exceeds bucket capacity ${this.maxTokens}`);
```

**Floating-point drift** — `elapsed * this.refillRate` accumulates small float errors over many calls. Negligible in practice for Observatory's use case, but worth knowing if you track exact token counts in assertions.

**No waiting / queue** — `tryConsume` is fire-and-forget. If callers need "retry after X ms," they'd need to compute `(cost - this.tokens) / this.refillRate * 1000` themselves. Consider exposing a `retryAfterMs()` helper if backpressure signaling becomes useful.

**In-memory only** — state resets on process restart. If Observatory restarts mid-window, all limiters start fresh. Fine for per-request limiting; not suitable for enforcing daily/monthly quotas without a persistent store.
