## Breakdown

### `_validation.ts` — Shared Input Guard

```ts
export function validateEmailAndName(
  email: unknown,
  name: unknown
): ValidationError | null
```

The underscore prefix signals this file is not an HTTP route — a SvelteKit/Next.js convention for co-located private modules. It accepts `unknown` inputs (coming from `req.body`, which could be anything) and performs three sequential guards per field:

- **email**: truthy → string type → contains `@`
- **name**: truthy → string type → length ≥ `MIN_NAME_LENGTH` (2)

Returns a typed `{ status: 400; error: string }` on first failure, or `null` on success. The literal `400` in the type (`status: 400`) is a discriminated union discriminant — callers can check `validationError.status` without an `instanceof` check.

The `MIN_NAME_LENGTH = 2` constant at module scope (rather than inlined) signals intent to make the threshold configurable without hunting for magic numbers.

### `invites.ts` and `users.ts` — Route Handlers

Both handlers are identical in structure:

```ts
const validationError = validateEmailAndName(req.body.email, req.body.name);
if (validationError) return validationError;
// ... domain logic
```

This is a **guard clause** pattern — validate at the top, exit early on failure, write the happy path without nesting. The shared `_validation` module means both routes enforce the same rules without duplication.

### `api-client.ts` — Outbound HTTP Wrapper

Two generic functions wrapping the Fetch API:

```ts
fetchData<T>(url: string): Promise<T>       // GET
saveData<T>(url: string, payload: unknown): Promise<T>  // POST
```

Both throw descriptive errors on non-OK responses (`GET ${url} failed: 404 Not Found`) — much better than letting `response.json()` fail silently or returning undefined.

`saveData` handles a subtle edge: a `204 No Content` response has no body, so calling `.json()` on it would throw. The check:

```ts
if (response.status === 204 || response.headers.get("content-length") === "0") {
  return undefined as unknown as T;
}
```

handles this gracefully. The `as unknown as T` double cast is a typed escape hatch — the caller accepts `T` may be `void` when the endpoint signals no content.

### `validator.ts` — Domain User Validation

A more granular validator than `_validation.ts`, focused on a richer `User` domain object:

```ts
function validateUser(user: User, settings?: Settings): ValidationResult
```

Four sequential guard clauses check:
1. User object exists
2. Email field present
3. Email is syntactically valid
4. If `settings.requireVerified` is set, user must be verified

Returns a `ValidationResult` discriminated union (`{ valid: true, user }` or `{ valid: false, error }`). The optional `Settings` parameter makes the verification check opt-in — callers that don't pass settings never fail on unverified users.

**Note:** This duplicates some email validation logic from `_validation.ts`. In a real codebase you'd unify these; here they serve different layers (HTTP boundary vs. domain logic).

### `processor.py` — Tax Calculation

```python
_TAX_MULTIPLIERS: dict[int, float] = {1: 1.08, 2: 1.15}

def calculate_total(items: list[dict], discount: float | None = None) -> float
```

The leading underscore on `_TAX_MULTIPLIERS` signals a module-private constant (Python convention). The function sums across items using a generator expression:

```python
total = sum(
    item["val"] * _TAX_MULTIPLIERS.get(item["type"], 1.0)
    for item in items
)
```

`.get(item["type"], 1.0)` is the key pattern — unknown item types default to a `1.0` multiplier (no tax) rather than raising a `KeyError`. This is a **safe default with open extension**: add a new type to `_TAX_MULTIPLIERS` and it activates without touching `calculate_total`.

## Patterns

| Pattern | Where | Notes |
|---|---|---|
| Guard clauses / early return | All route handlers, validators | Avoids deep nesting on validation branches |
| Discriminated union return types | `_validation.ts`, `validator.ts` | Type-safe error handling without exceptions |
| Generic fetch wrappers | `api-client.ts` | Single place to set headers, check status, and handle empty responses |
| Module-private constants | `_validation.ts`, `processor.py` | Separates "what the threshold is" from "how it's used" |
| Convention-based privacy | `_validation.ts` (underscore prefix) | Co-located private modules, not routes |

## Concerns

- **Weak email validation** — `email.includes("@")` rejects `foo` but accepts `@`, `a@`, `@b`, etc. Fine for a fixture; not for production.
- **`user` null check in `validator.ts`** — TypeScript knows `user: User`, so `if (!user)` is technically unreachable without a type cast at the call site. The check exists for runtime safety but the type signature promises a non-null input.
- **`processor.py` uses `item["val"]`** — KeyError will throw if an item dict is missing `val`. A `.get("val", 0)` default or a typed dataclass would be safer.
- **No error boundary in routes** — `handleCreateInvite`/`handleCreateUser` have no try/catch around the domain logic stub. In a real handler this would let unhandled exceptions propagate as 500s.
