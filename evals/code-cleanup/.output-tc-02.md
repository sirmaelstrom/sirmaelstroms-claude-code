## Findings

**High severity:**

1. **`any` types on both parameters** — All type safety is lost. `user` and `settings` should be typed interfaces.
2. **Pyramid of doom — 6 levels of nesting** — Classic arrow anti-pattern. Every failure case is an else-branch at the bottom of a deep stack. Guard clauses (early returns) eliminate this entirely.

**Medium severity:**

3. **Three identical success returns** — `{ valid: true, user: user }` appears three times (no settings, settings without requireVerified, settings with requireVerified and user.verified). Logic consolidation removes all duplicates.
4. **No return type annotation** — Callers can't see the contract without reading the body.

**Low severity:**

5. **Object longhand** — `{ user: user }` should be `{ user }` in modern TypeScript.

---

## Refactored

```typescript
interface User {
  email: string;
  verified?: boolean;
}

interface ValidationSettings {
  requireVerified?: boolean;
}

interface ValidationResult {
  valid: boolean;
  user?: User;
  error?: string;
}

function validateUser(user: User | null | undefined, settings?: ValidationSettings): ValidationResult {
  if (!user) return { valid: false, error: 'User required' };
  if (!user.email) return { valid: false, error: 'Email required' };
  if (!user.email.includes('@')) return { valid: false, error: 'Invalid email' };
  if (settings?.requireVerified && !user.verified) return { valid: false, error: 'User not verified' };
  return { valid: true, user };
}
```

---

## Summary

| Change | Benefit |
|---|---|
| Replaced `any` with typed interfaces | Callers get autocomplete, compiler catches misuse |
| Flattened 6 → 0 levels of nesting | Reads top-to-bottom; failures are immediately visible |
| Collapsed 3 duplicate success returns to 1 | Single source of truth for the happy path |
| Added explicit return type | Contract is visible without reading the body |
| `{ user }` shorthand | Idiomatic TypeScript |

Behavior is identical — the guard clause order maps exactly to the original nesting order. The `settings?.requireVerified` optional chain correctly handles both the `!settings` and `!settings.requireVerified` branches that were previously separate `if`/`else` levels.

**Further recommendation:** `includes('@')` is a very weak email check. For heathdev-observatory, a regex like `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` or a library like `validator.js` would be more appropriate at a real validation boundary — but that's a behavior change, so it's out of scope here.
